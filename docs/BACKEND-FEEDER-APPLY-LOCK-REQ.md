# 后端排查：申请喂养员接口误报「已提交」但后台无记录

> 状态：待后端排查 + 清理（前端已修复报错展示，但无法解决"提交被拒"本身）

## 现象
小程序申请成为喂养员，调用 `POST /api/miniapp/feeders`，某用户返回：
```
{code: 500, message: "您已提交过申请，请等待审核", data: null}
```
但**管理后台 feeder 列表看不到**该用户的申请记录。

## 根因推断
- 接口判重逻辑认为该 `userId` 已存在 feeder 记录，故拒绝。
- 该记录可能：
  1. `status` 不在后台列表查询的筛选范围内（如 `NULL` / 未定义值 / 被逻辑删除标记）；
  2. `user_id` 绑定异常（记录挂在别的账号下，当前登录用户查不到）；
  3. 早期测试产生的脏数据（开发 / 联调时提交过）。
- 结果：前端反复提交被挡，后台又看不到 → 用户卡死。

## 后端需执行的排查（按顺序）
1. 用触发用户的 **userId（或 phone）** 查 feeder 表，贴出该记录的：
   `id` / `user_id` / `status` / `created_at` / `phone`（如有）
2. 贴出**后台 feeder 列表查询**的 SQL / WHERE 条件，确认为何过滤掉该记录（是否漏了某个 status）。
3. 确认该记录是**测试脏数据**还是**真实重复提交**。
4. 确认接口判重是按 **phone 还是 userId**，是否与登录态一致（避免未登录 / 匿名请求被误判为"已提交"）。

## 处理方案（二选一或组合）
- **A. 清理脏数据**（推荐，最快恢复）：
  - 若为测试垃圾 → 直接 `DELETE` 该记录，或 `UPDATE` 其 `status` 为后台可显示的值。
  - 清理后用户可重新提交申请。
- **B. 支持「重新申请」机制**（长期）：
  - 提供撤回 / 覆盖接口（如 `DELETE /api/miniapp/feeders/me`，或允许 `status=REJECTED` 后重新提交）。
  - 接口判重时若记录为 `REJECTED`，允许覆盖而非拒绝。

## 前端已落地（供对齐）
- `apply.js` `doSubmit` 调 `feederApi.apply(form)` → `POST /api/miniapp/feeders`。
- 已修复（commit `d4e23a6`）：catch 不再静默吞错，业务错误由 `request.js` 原样 toast `message`。
- 本轮修正：遇到"已提交"错误时**不再**自动标记 `feederApplied=true`（避免首页误显"审核中"），改为弹窗提示"记录可能异常，联系管理员"。
- 接口契约未变，纯后端数据 / 逻辑问题。

## 验收
1. 后端清理该用户脏数据后，前端重新提交 → 返回成功，后台可见新申请。
2. （若做方案 B）已 `REJECTED` 的用户可重新提交，不再被"已提交"挡住。
3. 直接 curl 绕过前端验证清理逻辑一致性。
