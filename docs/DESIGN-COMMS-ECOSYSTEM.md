# 沟通 / 消息生态整体设计 — 毛茸茸暖屋

> 设计定位：沟通**不是独立功能**，而是「把毛孩子托付给陌生人」这件事里建立信任的核心环节。
> 它必须无缝嵌进「浏览喂养员 → 沟通 → 下单 → 报价 → 确认 → 服务 → 评价」全链路，而不是一个孤立的聊天框。
> 本文是 UI Designer 对该生态的**整体设计定稿**，视觉规范复用 `DESIGN.md`（奶油暖屋体系）。

---

## 1. 信息架构（IA）

### 底部四 Tab（已落地）
`首页` → `消息` → `订单` → `我的`

- **消息 Tab = 聊天总入口**：会话列表，承载所有一对一沟通。
- 选「消息」而非「沟通」作 Tab 名，是因为它本质是一个**消息中心**（主人+喂养员双向），且符合用户对底部栏「消息」的心智。

### 沟通入口矩阵（4 个入口，全部已落地前端）
| # | 入口位置 | 触发方式 | 跳转参数 | 适用角色 |
|---|----------|----------|----------|----------|
| ① | 喂养员列表卡片右侧圆形的「💬 沟通」 | `catchtap` 不触发进详情 | `?feederId={id}` | 主人 |
| ② | 喂养员详情页底部「💬 沟通」按钮 | 与「去预约」并排 | `?feederId={id}` | 主人 |
| ③ | 订单详情页「联系喂养员」按钮 | 有 feederId 时显示 | `?orderId={id}` | 主人 / 喂养员 |
| ④ | 消息 Tab 会话列表点进 | 默认 | `?conversationId={id}` | 双向 |

**设计约束**
- 仅**主人**可主动发起沟通（①②）；喂养员/管理员不与自己聊（后端 `by-feeder` 对 `feederId==自己` 返回 400）。
- 会话详情页 `chat/detail` **三参数互斥优先**：`conversationId` > `orderId` > `feederId`，后端按优先级取/建会话。
- 同一对 `owner+feeder` **仅一个会话**（`UNIQUE(ownerId, feederId)`），无论是否绑定订单——避免反复开新窗。

## 2. 核心用户流（沟通嵌入交易闭环）

```
[浏览喂养员]
   │  ①/② 可直接「💬 沟通」← 不强制先下单
   ▼
[沟通议价] ─────────────┐ 问价格/服务细节/是否含猫砂
   │                      │
   ▼                      │
[下单] 客户填需求(无价)    │ ③ 订单详情「联系喂养员」持续沟通
   │                      │
   ▼                      │
[喂养员报价] PENDING→QUOTED
   │                      │
   ▼                      │
[客户确认] QUOTED→ACCEPTED
   │                      │
   ▼                      │
[喂养员服务] ACCEPTED→IN_PROGRESS ← 服务中仍可沟通进度
   │                      │
   ▼                      │
[完成] IN_PROGRESS→COMPLETED
   │
   ▼
[评价] 完成后可评
```

**关键设计决策（UI Designer 拍板，无需再确认）**
- 沟通在**浏览期、报价后、服务中**三个时点都可发生，不是只有下单后。
- 价格沟通以「结构化报价→接受/拒绝」为主干，**自由聊天为补充**（聊天里能说"80 能优惠吗"，喂养员回"70 含猫砂"，但最后仍回到订单的确认动作锁价）。
- 确认报价即锁定价格、进入服务流程；**支付暂未接入**（后续如需要再评估平台托管等方案）。

## 3. 消息模块交互规范

### 3.1 会话列表 `pages/chat/list`
- **加载态**：骨架屏（3 张占位卡）。
- **空态**：大 💬 + 「还没有沟通记录」+ 引导文案 + **「去找喂养员聊聊 ›」按钮**（跳喂养员列表，主动破冰）。
- **列表项**：圆形头像/首字母 → 名字 + 「认证」绿标 → 最后消息（单行截断）→ 时间（右侧弱灰）→ 未读红点（暖橙胶囊，>99 显 `99+`）。
- **TabBar 红点**：进页拉取后汇总所有会话未读，`wx.setTabBarBadge` 显示总数；全部已读则 `removeTabBarBadge`。每次 `onShow` 刷新（从会话返回即更新）。

### 3.2 会话详情 `pages/chat/detail`
- **气泡**：左侧对方=白卡 + 暖墨字；右侧自己=暖橙渐变 + 白字；圆角 24rpx，最大宽度 70%。
- **时间**：同天显 `HH:mm`、跨天显 `MM/DD`，弱灰小字。
- **发送态**：`发送中`（灰字）→ `已送达`；失败显「重发」可点重试（temp id 机制）。
- **实时**：进页后 8s 轮询拉增量（不依赖 WebSocket，省心；后续可换长连接）。
- **已读**：进页即 `markRead`，并清 TabBar 红点。
- **输入栏**：吸底 + `safe-bottom` 安全区；暖橙胶囊「发送」按钮，空内容置灰。

### 3.3 视觉（复用 DESIGN.md）
| 元素 | 规范 |
|------|------|
| 自己气泡 | `linear-gradient(135deg,#FB923C,#F97316)` + 白字 + 暖橙柔光阴影 |
| 对方气泡 | `#FFFFFF` + `--wf-ink` + `1rpx solid #FFF3E8` |
| 未读红点 / TabBar badge | `--wf-orange (#F97316)` |
| 时间 / 辅助 | `--wf-ink-faint (#B8A99C)` |
| 输入栏 | 白底 + `1rpx solid --wf-border` + 吸底柔光 |

## 4. 状态机

**会话**：不存在 → 由入口触发创建 → 有未读(n) / 已读(0) ⇄ 消息进出切换。
**消息**：`sending` → `sent`（失败 → `failed` → 用户 `retry` → `sending`）。
**订单（沟通上下文）**：`PENDING→QUOTED→ACCEPTED→IN_PROGRESS→COMPLETED`（聊天不阻断状态机，只是伴随）。
