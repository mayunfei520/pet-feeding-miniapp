# 后台管理列表刷新机制需求（前端 UX 规范）

> 关联 Issue：pet-feeding-admin #8
> 范围：**仅**管理后台（Vue 前端）列表刷新体验，不涉及小程序端，也不涉及接口数据正确性。

## 背景
用户在管理后台「喂养员管理」界面观察到：切到「全部 / 已认证」标签后，列表需要**等一会儿、或手动刷新**才能加载出记录；审核 / 认证等操作成功后，列表也不立即更新。给人“数据延迟”的错觉。

补充澄清：用户之前在「待审核」看不到自己的记录、去「全部 / 已认证」才看到，这是**状态分类正确**（其 feeder 记录已是“已认证”，不在待审核队列），并非数据丢失。本需求聚焦的不是“数据在哪”，而是“**列表展示为何不能及时刷新**”。

## 根因推断（前端视角）
后台列表组件很可能仅在**页面首次挂载时拉取一次**数据，之后：

- 切换 Tab（待审核 / 已认证 / 全部）未重新请求该 Tab 数据，残留旧列表或空白；
- 审核 / 认证等操作成功后只弹 toast，未自动 `re-fetch` 当前列表；
- 无手动刷新入口、无轮询、无可感知的加载态。

## UX 规范要求（请后台前端落实）

1. **手动刷新按钮**
   列表区右上角常驻「↻ 刷新」图标 / 按钮，点击立即 `re-fetch` 当前 Tab 数据。

2. **操作后自动刷新**
   任何「通过 / 拒绝 / 认证」等写操作 `success` 回调里，自动 `re-fetch` 当前列表（不要只 toast 不刷新）。

3. **Tab 切换即刷新**
   切到任一状态 Tab 时立即加载该 Tab 数据，不缓存 / 不残留上一 Tab 内容。

4. **加载态可见**
   `fetch` 期间显示骨架屏或 spinner，列表区短暂禁用交互，让用户明确感知“正在加载”。

5. **空状态明确**
   某 Tab 无数据时显示「暂无 XXX 记录」，杜绝空白导致的“是不是没加载”困惑。

6. **（可选）轻轮询**
   对「待审核」等需实时性的队列，可每 15–30s 静默轮询一次（不阻断用户操作、不打扰），确保新提交及时出现。

## 验收标准

- [ ] 在「待审核」通过一条 → 列表自动跳到「已认证」且**立即**出现该记录，无需手动刷新。
- [ ] 任意 Tab 间切换，不残留上一 Tab 的旧数据。
- [ ] 点击「刷新」按钮，1s 内当前列表更新。
- [ ] 操作后 1s 内列表反映最新状态。
- [ ] 某状态无数据时，显示明确空状态文案而非空白。

## 设计原则（UI Designer 视角）

- **可感知性**：任何异步数据获取都必须有可见的加载 / 刷新反馈，避免用户以为“卡了”或“没数据”。
- **操作即反馈**：状态变更类操作（通过 / 拒绝 / 认证）成功后应即时反映在当前视图，而非依赖用户手动刷新去“发现”变化。
- **一致性**：所有列表页（喂养员管理、订单管理等）统一采用同一套刷新机制，避免体验碎片化。
